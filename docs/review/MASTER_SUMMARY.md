## 二、跨评审员**高置信共识问题**（2 人以上指出）

这些是最应该优先处理的问题，多位评审员独立发现 → 可信度极高。

### 共识 #1：Property<T> propHeader 机制过度超前（4 位评审员共识）

| 评审员 | 具体指控 |
|-----|------|
| 架构员 | "hasExt 本期固定写 0，Decoder 也不读；extHeader 尺寸冻结为 1 byte 形成隐式 ABI" |
| 精简员 | "D-6: §4.3 Property 编码重复段 53 行" |
| 扩展员 | "hasExt 为一个未使用的扩展点冻结设计" + "bit 6-7 永久冻结过度防守" |
| 维护员 | "Property<T> 编码规则在 §4.3/§5.2/§13.1 三处重复定义，缺单一真相源" |

**共识结论**：**hasExt 位 + bit 6-7 永久冻结约束应当删除**。实际扩展时直接升 FORMAT_VERSION。

**预期收益**：节省 ~80 行文档 + 消除"隐式 ABI 冻结"心智负担 + 每个 Property 节省潜在 1 byte

---

### 共识 #2：Tag 版本化机制（三层兼容中的第②层）过度超前（3 位评审员共识）

**预期收益**：节省 ~150 行规范文本 + 消除"LIFO 双写优先级"规约 + 扩展心智模型从 3 层简化为 2 层

## 六、评审员分歧点

### 分歧 #1：Path 双格式是否保留

- **维护员**：必须补全（AI 落代码关键）
- **精简员**：未提及

### 分歧 #3：FORMAT_VERSION "5 年不升"承诺
- **扩展员**：建议删除承诺
- **其他**：未提及
- **主决策者倾向**：**删除承诺**。未发布的协议不应预支未来决策权
