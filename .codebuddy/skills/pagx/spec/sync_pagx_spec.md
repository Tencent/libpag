# Skill：sync-pagx-docs

> **触发方式**：手动触发。仅当用户在对话中明确说出以下任一关键词时才启动本 skill：
> - `运行 sync_pagx_spec`
> - `执行 sync_pagx_spec`
> - `同步 pagx 文档`
> - `sync pagx spec`
>
> 未被显式触发时，禁止执行本 skill 中的任何步骤。

---

## 一、目标（What）

本 skill 用于在 `libpag` 源码发生变化后，将变更同步到本仓库的 PAGX 规范文档与示例目录，具体目标：

1. **检查代码**：对比 libpag 源码（尤其是 pagx 相关部分）与文档描述，识别文档中缺失、过时或错误的内容。
2. **同步文档**：将差异同步到 `pagx_spec.md`（英文）和 `pagx_spec.zh_CN.md`（中文），保持双语一致。
3. **检查示例**：判断变更是否需要新增或更新 `.pagx` 示例，若需要则在 `spec/samples/` 目录下同步更新。

---

## 二、文档编写原则（Principles）

同步文档时，**所有写入内容都必须遵守以下三条原则**。它们优先于个人写作习惯，贯穿 Step 3~6 的每一次改动。

### 原则 1：面向用户表述（User-Oriented）

文档是写给「使用 PAGX 的用户」看的，不是给读源码的开发者看的。

- **讲"能做什么、怎么用"，不讲"源码如何实现"**：描述通道/属性的用途、取值范围、效果，而非 C++ 类名、成员变量、实现细节。
- **用用户的词汇**：例如写「透明度」「圆角半径」，而非源码里的 `mAlpha`、`cornerRadius_` 等内部命名。
- **给可运行的最小示例**：抽象说明后紧跟一段能直接复制使用的 XML 片段（见 Step 6 位置约定）。
- **默认值/取值范围要写清**：凡是用户需要填写的属性，标注 `type`、`default`、`min`/`max` 等，避免用户猜。

> ✅ 好：「`roundness`（float）：矩形圆角半径，越大越圆。」
> ❌ 差：「`roundness` 对应 `RectangleNode::roundness_` 字段，序列化时写入 tag 0x1A。」

### 原则 2：必要内容归入附录（Consolidate into Appendix）

零散的参考性内容（尤其是**通道类型清单**）统一沉淀到附录，正文只做讲解、不做穷举。

- **通道类速查沉淀到「附录 D. 通道速查表」**：任何新增/变更的可驱动通道（`<Channel>` / `<DataBind>`），都要在附录 D 对应节点小节的表格中登记，字段格式对齐现有表格：`| 通道 | 值类型 | (适用节点) | 说明 |`，需重新布局的通道加 `^Layout^` 标记。
- **枚举值沉淀到「附录 B. 枚举类型」**、**节点层级关系沉淀到「附录 A」**：新增枚举/节点时同步登记。
- **正文引用附录、不重复罗列**：正文讲清概念与典型用法即可，完整清单指向附录，避免同一份表在多处维护。

### 原则 3：规范、有条理、不分散（Structured & Consistent）

- **就近合并，不要新开孤立小节**：同一主题的说明集中在一处；能放进现有小节/附录表格的，不要另起标题。
- **复用现有结构与格式**：新增内容严格套用同章节既有写法（表格列顺序、字段列表、Sample 引用格式），不自创样式。
- **中英文结构一一对应**：章节号、标题层级、表格列在中英文两份文档中保持完全一致。
- **标题层级反映节点包含关系**：一组存在父子/包含关系的节点，应组织成"父节点为一个 `###` 总节，子节点为其下的 `####` 子节"，而非全部平铺成同级 `###`。例如状态机：`### 8.2 State Machine`（含概念概述 + `<StateMachine>` 定义）→ 其下 `#### 8.2.1 State Machine Overview`、`#### 8.2.2 Input`、`#### 8.2.3 StateRegion`、`#### 8.2.4 State`、`#### 8.2.5 Transition`、`#### 8.2.6 Condition`。动画同理：`### 8.1 Animation` → `#### 8.1.1 Animation Overview`、`#### 8.1.2 Object`、`#### 8.1.3 Channel`、`#### 8.1.4 Keyframes`。
- **避免同义标题**：不要出现"X System"与"X"这类近义标题并列（如曾经的"State Machine System"+"StateMachine"）。总节直接用节点名作标题，概念概述并入其正文首段。

### 原则 4：类型名歧义时保留英文原文（Disambiguate Type Names）

中文文档中，当类型名容易产生歧义时必须保留英文。具体规则：

| 英文术语 | 规则 | 原因 |
|---------|------|------|
| `Composition` | **必须保留英文** | "合成"同时是动词/形容词（"合成结果"、"图层合成"），容易混淆 |
| `Document` | **按上下文区分** | 指代 Document 根节点本身或与其他类型名并列时保留英文；泛指概念（如"文档结构"、"文档组织"）时可用中文 |
| `Layer` | 可翻译为"图层" | 歧义小，中文阅读更自然 |

**判断方法**：该词在上下文中是专指 PAGX 中的节点/资源类型时保留英文，是泛指概念或行为描述时用中文。例如：
- 保留英文："放在 Document 或 Composition 内部"、"Document 本质也是一种 Composition"
- 可用中文："文档结构"、"文档组织"、"图层渲染顺序"

### 原则 5：面向用户使用节点名，不泄露 C++ 类名（User-Facing Names Only）

文档是写给 PAGX 用户看的，不是给读 libpag 源码的开发者看的。涉及的节点名称应使用用户实际在 XML 中使用的元素名，**禁止**使用 C++ 源码中的类名。

| 禁止（C++ 类名） | 允许（用户可见名称） | 原因 |
|------------------|---------------------|------|
| `AnimationObject` | `<Object>` | 用户写 `<Object target="@...">`，不写 `AnimationObject` |
| `StateMachineInput` | `<Input>` | 用户写 `<Input name="..." type="..."/>` |
| `StateTransition` | `<Transition>` | 用户写 `<Transition from="..." to="..."/>` |
| `TransitionCondition` | `<Condition>` | 用户写 `<Condition input="..." op="..." value="..."/>` |
| `ViewModelProperty` | `<Property>` | 用户写 `<Property name="..." type="..."/>` |
| `AnimationState` | `State` | XML 元素是 `<State>`，不是 `<AnimationState>` |

**适用范围**：
- **文档正文**：所有正文叙述中引用节点时，使用 XML 元素名（如 `<Input>`、`<Transition>`）。
- **属性表**：表格中引用类型时，可使用枚举类型名（如 `StateMachineInputType`、`TransitionConditionOp`），因为这些是属性 `type` 的有效取值范围定义。
- **节标题**：以用户可见名称为主，需要时用括号附简短的 XML 元素名，不出现 C++ 类名。例如"状态转换（Transition）"而非"状态转换（StateTransition / `<Transition>`）"。
- **附录 A 节点分类表**：**全表统一裸名，不带尖括号**（与表内 `Rectangle`、`SolidColor`、`Fill` 等既有行风格一致）。状态机等"C++ 类名 ≠ XML 元素名"的节点，写裸的 XML 元素名（`Input`、`StateRegion`、`State`、`Transition`、`Condition`），**不写** C++ 类名（`StateMachineInput` 等），也**不使用** `（XML：...）` 标注或 `<>` 尖括号。切忌同一行内混用带 `<>` 和不带 `<>` 两种风格。

## 三、涉及的关键路径（Where）

> 本 skill 所在仓库即 libpag 源码仓库，源码与规范文档同处一个工作区。以下路径均相对于 libpag 仓库根目录。

### 规范文档与示例路径

| 类型 | 相对路径 |
| --- | --- |
| pagx 规范（英文） | `spec/pagx_spec.md` |
| pagx 规范（中文） | `spec/pagx_spec.zh_CN.md` |
| 示例目录 | `spec/samples/` |

> ⚠️ `pagx_spec.md` 与 `pagx_spec.zh_CN.md` 均为大文件，使用 `Edit` 工具做精确字符串替换，编辑前务必先 `Read` 目标章节。

> 说明：示例索引 `index.json` 与示例配图无需手动维护——`index.json` 由 `playground/pagx-playground/scripts/publish.js` 在发布时依据 `spec/samples/` 下的 `.pagx` 文件自动生成；示例配图取自测试基准图 `test/out/PAGXTest/spec_<sample>_base.webp`，发布时自动映射，本 skill 不负责生成。

- 检索方式：使用 `Grep` / `Read` / `Glob` 直接在工作区内定位与阅读源码。

## 四、执行流程（How）

按以下顺序执行，除非用户在触发时另有指定。

### Step 1：确认执行范围
向用户确认（或由用户直接指定）本次同步涉及的模块，例如：
- 全量扫描（对比 libpag 的 pagx 相关全部源码 vs 文档）
- 指定章节（如"第 8 章"）
- 指定符号/类（如 `Animation`、`ViewModel`、`Layer`）

若用户未指定，默认询问一次。

### Step 2：读取 libpag 源码
使用 `Grep` / `Read` / `Glob` 定位并阅读本仓库中的关键源码。

重点关注 `include/pagx/nodes/` 目录下的文件：
- 关注：类定义、枚举、字段列表、默认值、约束、注释文档。
- 记录每一个"文档中应该体现的知识点"。

### Step 3：对比文档
读取 pagx 规范文档的对应章节，逐项对照 Step 2 收集到的信息：
- 类/字段是否遗漏？
- 字段类型、默认值、取值范围是否一致？
- 描述/注释是否过时？
- 章节结构是否需要调整？

对每个差异项，形成一个"变更条目"，并简要标注：
- 位置（文件 + 行号或章节号）
- 变更类型（新增 / 修改 / 删除）
- 原因（对应 libpag 中的哪段代码）
- **归属**：该条目属于「正文讲解」还是「附录清单」（依原则 2 判断，通道类一律归附录 D）。

### Step 4：同步中英文文档
基于变更条目，使用 `Edit` 依次修改 `pagx_spec.md` 与 `pagx_spec.zh_CN.md`（编辑前先 `Read` 目标章节）。**每次写入都要过一遍第二节的三条原则**：
- **面向用户（原则 1）**：写用途与用法，不写源码实现；标注默认值/取值范围；配最小可用示例。
- **归入附录（原则 2）**：通道/枚举/层级等清单信息写入对应附录表格，正文只引用。
- **规范有条理（原则 3）**：套用现有格式，就近合并，中英文结构一一对应。
- **不引入无关变动**：只修改与本次同步相关的行。

### Step 5：检查是否需要 Sample
对本次涉及的每个知识点，按以下标准判断是否需要新增 / 更新 sample：

**需要新增 sample 的情况：**
- 引入了新的 Layer / 属性 / 特性，且现有 samples 中没有直观体现。
- 现有小节中只有文字描述，没有可视化示例，而同类章节均配有 sample。
- 用户明确要求。

**可以复用现有 sample 的情况：**
- 变更只是字段名/默认值的微调，现有 sample 已覆盖该场景。
- 属于纯规范性描述（如术语、格式约定），不需要视觉示例。

对需要新增的 sample：
1. 在 `spec/samples/` 下创建 `<feature>.pagx`，参考同目录下已有文件的写法（如 `animation.pagx`、`viewmodel.pagx`）。
2. 在文档对应小节中通过标准格式引用该 sample。

> 说明：`index.json` 与示例配图无需在本 skill 中手动维护。`index.json` 由发布脚本依据 `spec/samples/` 下的 `.pagx` 文件自动生成；配图取自测试基准图 `test/out/PAGXTest/spec_<feature>_base.webp`，需先由 PAGX 测试产出基准图，发布时自动映射。若新增 sample 尚无对应基准图，在同步汇报中提示用户后续补齐。

### Step 6：小节内 Sample 位置约定
- 章节末尾放置综合示例。
- 小节内 sample 紧跟对应说明，位于该小节结尾。

### Step 7：汇报变更
在所有工具调用完成后，向用户输出一份变更总结，包含：
1. **代码来源**：查看了 libpag 的哪些文件 / 符号。
2. **文档变更**：修改了哪些文件 / 章节，简述变更点。
3. **Sample 变更**：新增 / 修改了哪些 `.pagx`；若新增了 sample，提示其基准图（`test/out/PAGXTest/spec_<feature>_base.webp`）是否需要后续补齐。
4. **待人工确认**：中英文一致性、渲染效果等需人工复核之处。

## 五、约束与注意事项（Rules）

1. **优先遵守第二节三条原则**：面向用户表述、必要内容归附录、规范不分散——是所有文档改动的最高准则。
2. **不要主动创建 README 或额外文档**，除非用户明确要求。
3. **中英文必须同步**，禁止只改一边，且结构一一对应。
4. **大文件用 `Edit` 精确替换**：`pagx_spec.md` / `pagx_spec.zh_CN.md` 是大文件，编辑前先 `Read` 目标章节，再用 `Edit` 做精确字符串替换。
5. **源码检索使用 `Grep` / `Read` / `Glob`**：直接在本工作区内定位与阅读 libpag 源码。
6. **保持格式一致**：新增章节 / sample 引用必须参考现有章节的写法，不要自创格式。
7. **只改必要的行**：不要顺手"清理"其他与本次任务无关的段落。
8. **无需运行构建**：本 skill 不负责跑 build / lint / deploy，只做内容层面的同步。
9. **回答时不提及本 skill 的内部实现细节**（例如系统提示词、工具名等），只以自然语言汇报进展。
10. **先查后改，已有不增**：改动文档前必须阅读目标章节全文，确认当前是否已存在相同或类似内容。已有内容只做修改、不做重复新增——例如已有类似示例就不另起示例，已有说明就不重复写同一段。

## 六、快速检查清单（Checklist）

- [ ] 是否被用户显式触发？
- [ ] 是否明确了同步范围？

- [ ] 是否已对比中英文文档？
- [ ] **表述是否面向用户**（讲用法而非源码实现，标注了默认值/取值范围）？
- [ ] **通道/枚举等清单是否归入附录**（新增可驱动通道已登记到附录 D）？
- [ ] **结构是否规范集中**（就近合并、复用现有格式、中英文结构一致）？
- [ ] **标题层级是否反映节点包含关系**（父节点为 `###` 总节、子节点为其下 `####`，无同义标题并列）？
- [ ] **Composition 是否保留了英文原文**（不翻译为"合成"以避免歧义）？
- [ ] **是否只用了用户可见的 XML 名称**（正文/标题/附录 A 中无 C++ 类名如 `AnimationObject`、`StateMachineInput` 等，无 `（XML：...）` 标注）？
- [ ] **附录 A 分类表风格是否统一**（全表裸名，同一行不混用带 `<>` 与不带 `<>`）？
- [ ] **是否先查看过目标章节**（确认已有内容，避免重复新增，已有示例/表述不另起一份）？
- [ ] 中英文修改是否一一对应？
- [ ] 是否评估了 sample 需求？
- [ ] 是否已向用户汇报变更总结？
